APPLIED AI JOURNEY SUMMARY (MANOJ)
LangChain → RAG → Memory → LCEL → System Thinking

========================================================
1. PHASE 1: FOUNDATIONS (LLM PIPELINES)
========================================================

Goal:
Understand how LLM applications are built step-by-step.

Flow:

User Input
   ↓
Prompt Template (ChatPromptTemplate)
   ↓
LLM (ChatGroq / OpenAI)
   ↓
Output Parser (StrOutputParser)
   ↓
Final Response

Key Insight:
- LLM = reasoning engine
- Prompt = controls behavior
- Parser = cleans output

LCEL Pattern:
prompt | model | parser

========================================================
2. PHASE 2: DOCUMENTS + RAG (KNOWLEDGE SYSTEMS)
========================================================

Concept:
LLM alone is not enough → needs external knowledge

Flow:

Raw Data
   ↓
Document(page_content, metadata)
   ↓
Embeddings (HuggingFace / OpenAI)
   ↓
Vector DB (Chroma)
   ↓
Retriever
   ↓
LLM (with context)
   ↓
Answer

Key Insight:
- Retriever = fetch
- LLM = reason

RAG Chain:

{
  "context": retriever,
  "question": RunnablePassthrough()
}
   ↓
Prompt
   ↓
LLM

========================================================
3. PHASE 3: RUNNABLES (DATA PIPELINE THINKING)
========================================================

Concept:
Everything becomes a pipeline (LCEL)

Flow:

Input
   ↓
Transform (RunnableLambda)
   ↓
Pass-through (RunnablePassthrough)
   ↓
Prompt
   ↓
Model
   ↓
Output

Runnable Types:

RunnablePassthrough:
- passes input unchanged

RunnableLambda:
- custom logic

Key Insight:
- LCEL = data pipeline for AI systems

========================================================
4. PHASE 4: CHAT MEMORY (STATEFUL SYSTEMS)
========================================================

Concept:
LLMs are stateless → memory is external

Flow:

session_id
   ↓
ChatMessageHistory
   ↓
[Human, AI, Human, AI...]

RunnableWithMessageHistory:
   ↓
Fetch history
   ↓
Inject into prompt
   ↓
Run model
   ↓
Store new messages

Example:

chat1 → remembers conversation
chat2 → new conversation

Key Insight:
- No magic → just storing + re-sending messages

========================================================
5. PHASE 5: PROMPT + MEMORY INJECTION
========================================================

Prompt Structure:

System Message
+ MessagesPlaceholder("messages")
+ User Input

Flow:

messages (runtime)
   ↓
MessagesPlaceholder
   ↓
Prompt
   ↓
LLM

Key Insight:
- dynamic prompting
- flexible conversation injection

========================================================
6. PHASE 6: TOKEN & MEMORY MANAGEMENT
========================================================

Problem:
Too many messages → token overflow

Flow:

Full History
   ↓
Trim Messages (max_tokens)
   ↓
Reduced Context
   ↓
LLM

Observation:
- trimming can remove important info
- leads to forgetting

Key Insight:
- memory is limited
- must manage context size

========================================================
7. PHASE 7: CONFIG & SESSION CONTROL
========================================================

Concept:
Separate system control from input

Flow:

Input:
{"question": "..."}

Config:
{
  "configurable": {
    "session_id": "chat1"
  }
}

Key Insight:
- input = business logic
- config = system behavior
